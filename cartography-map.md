## backend/app/agents/

- **Owns:** The namespace that groups all agent submodules. The parent `__init__.py` is empty — it owns nothing directly. There is no `BaseAgent`, no shared input/output schema, and no pipeline coordinator at this level. Each agent is completely standalone.
- **Does NOT own:** Any shared base class, common interface, registry, or inter-agent orchestration logic.
- **Spotted contracts:** None exported. No shared type definitions, no env vars read here, no config.
- **Confidence:** HIGH
- **Node recommendation:** NEEDS NODE (see CONFLICT C1 below)
- **Reason:** While the parent package is empty, the cross-cutting observations from the agents cartographer found important shared patterns: all agents except `image_text_matching` route through `llm/gemini.py`; there is an async-vs-sync inconsistency across agents; there is no shared input/output contract. A node here documents the absence of shared infrastructure (which is itself architectural knowledge).

## backend/app/agents/image_extraction/

- **Owns:** Extracting high-quality, deduplicated keyframes from video files. Full 5-phase pipeline: (0) video inspection/duration, (1) uniform candidate timestamp generation, (2) frame extraction via OpenCV, (3) quality scoring/filtering (blur detection via Laplacian variance, eye-open detection via Haar cascades), (4) perceptual hash deduplication, (5) final frame selection with per-scene coverage. Also owns a self-contained DCT-based perceptual hash implementation (`phash.py`).
- **Does NOT own:** Image-text matching, any LLM calls, image upload or metadata storage. All output is written to a local filesystem path under `outputs/keyframes/`.
- **Spotted contracts:**
  - `run_keyframe_pipeline(video_path: str, config: Optional[Dict] = None) -> Dict[str, Any]` — main entry point
  - `DEFAULT_CONFIG` with tunable knobs: `sample_period_seconds=5.0`, `blur_threshold=30.0`, `max_total_frames=10`, `dedup_threshold=8`, `per_scene_target=2`
  - `detect_scenes()` — exported but dead from pipeline perspective (superseded by uniform sampling)
  - Dependencies: `cv2`, `numpy`, `Pillow`, `scenedetect` (lazy-imported for `detect_scenes` only)
- **Confidence:** HIGH
- **Node recommendation:** NEEDS NODE
- **Reason:** 4 files, a 5-phase stateful pipeline, non-trivial scoring model with magic weights, intentional design choice to switch from scene-detection to uniform sampling (making `scene_detection.py` dead code inside the pipeline), and a custom phash implementation.

## backend/app/agents/image_generation/

- **Confidence:** HIGH
- **Node recommendation:** NO NODE NEEDED
- **Reason:** 1 file, 2 thin wrapper functions. Hardcoded MIME type and duplicated model name are pitfalls worth noting in parent node.

## backend/app/agents/image_text_matching/

- **Owns:** Semantic matching of video keyframes to text summaries using Fireworks AI (Qwen 2.5 VL model) — NOT Gemini. 3-call-per-pair pipeline: (1) OCR/text extraction, (2) image captioning, (3) direct semantic similarity rating. Weighted `combined_score`. In-memory caching by `image_id`. Owns its own type definitions (`TextSummary`, `ImageCandidate`, `ImageMatch`).
- **Does NOT own:** Image extraction from video, Gemini routing. This is the only agent that calls a non-Google LLM. `ImageMatch.timestamp_score` is always `0.0` — structurally present but never computed.
- **Spotted contracts:**
  - `ImageTextMatcherVLM` — async context manager class
  - `VLMConfig` reads `FIREWORK_API_KEY`, `FIREWORKS_VLM_MODEL` from env. Default weights: `semantic=0.6`, `detail=0.4`
  - Detail score uses a word-overlap heuristic (not an LLM call)
  - PITFALL: async methods make synchronous blocking Fireworks API calls — the async interface is misleading
  - PITFALL: `FIREWORKS_VLM_MODEL` has no default fallback — `None` if env var absent, causing silent API errors
- **Confidence:** HIGH
- **Node recommendation:** NEEDS NODE
- **Reason:** 4 files, distinct external API (Fireworks, not Gemini), own type system, async-but-blocking pitfall, dead `timestamp_score` field, missing env var default.

## backend/app/agents/quote_extraction/

- **Owns:** Extracting verbatim pull-quotes from transcripts using Gemini structured output. 5 named styles (`general`, `punchy`, `insightful`, `contrarian`, `emotional`). Layered fallback: Gemini structured output → sentence-splitting fallback. Post-processing: JSON parsing, quote sanitization, length filtering, verbatim-in-transcript verification, deduplication, sentence-casing, punctuation enforcement.
- **Does NOT own:** Transcript sourcing, summarization, storage.
- **Spotted contracts:**
  - `extract_quotes(transcript, style, count) -> List[Dict[str, str]]` (async)
  - `STYLE_OPTIONS: set` — unknown styles silently normalize to `"general"`
  - `count` clamped to `max(1, min(count, 30))`
  - Style word-length bounds are magic numbers in `_style_length_bounds`
  - Verbatim check: normalizes both quote and transcript (lowered, punctuation stripped) — slightly rephrased quotes are filtered out
- **Confidence:** HIGH
- **Node recommendation:** NEEDS NODE (see CONFLICT C2 below)
- **Reason:** Although 1 file, the filtering/validation logic is dense (8 private helpers). The multi-layer fallback and verbatim-check filtering are non-obvious behaviors.

## backend/app/agents/text_generation/

- **Owns:** All text content generation. Two parallel architectures coexist: (1) `generator.py` — preset-driven path fetching prompt templates from Supabase `text_generation_presets` table, supports runtime overrides, calls `query_gemini` with optional structured output; (2) Four standalone generators — `email_generation.py`, `linkedin_generation.py`, `tiktok_generation.py`, `chatbot_generation.py` with hardcoded prompts. Also owns `content_parser.py` — post-processing parsers for email/LinkedIn/TikTok.
- **Does NOT own:** Storing generated content, scheduling posts. `content_parser.py` functions are NOT called internally — they are orphaned utilities for external callers. `parse_tiktok_content` hardcodes placeholder `@micra_official` and `Original Sound - MiCRA`.
- **Spotted contracts:**
  - `generate_text(input_text, preset_id, ...) -> Dict[str, Any]` — primary unified path
  - `generate_email(source_texts?, tone_preference?) -> str`, `generate_linkedin_post(...)`, `generate_tiktok_script(...)`, `generate_chatbot_response(...)` — standalone paths returning raw strings
  - All standalone generators take `source_texts: Optional[List[Dict]]` where each dict must have `'title'` and `'content'` keys
  - `{source_context}` placeholder takes precedence over `{input_text}` in templates — non-obvious priority order
  - Regex guard strips trailing "Visual Suggestions" sections from freeform output
- **Confidence:** HIGH
- **Node recommendation:** NEEDS NODE
- **Reason:** 6 files with two parallel generation architectures (preset-driven vs. hardcoded). Callers must know which path they're on. Orphaned `content_parser.py`, duplicate `source_context` building logic across all modules.

## backend/app/agents/video_generation/

- **Owns:** AI video generation via Google Veo 3.1, plus multi-step preprocessing. Preprocessing (`preprocessor.py`): image selection (max 3), per-image Gemini analysis, prompt enhancement. Generator (`generator.py`): auth mode detection (Vertex AI vs Gemini API key rotation), Veo API submission, long-poll waiting loop (10s intervals, 300s max), multi-strategy video byte extraction.
- **Does NOT own:** Audio, transcription, frame extraction. Video inputs are accepted but "not currently used by Veo" — dead input handling. Gated by `VEO_ENABLE_LIVE_CALLS=true` — without it every call raises `RuntimeError`.
- **Spotted contracts:**
  - `generate_video_with_veo(prompt, images=None, params=None) -> bytes` — returns raw MP4 bytes
  - Reference images force `duration_seconds` to `"8"` (Veo 3.1 limitation)
  - Hardcoded: `MODEL = "veo-3.1-generate-preview"`, default project `"core-avenue-488216-t2"`, location `"us-central1"`
  - `preprocess_video_inputs(params, inputs) -> Tuple[str, list[bytes], dict]`
  - `VIDEO_STYLE_DIRECTIVES` — 5 preset style strings (extensibility point)
  - Env vars: `VEO_ENABLE_LIVE_CALLS`, `GOOGLE_APPLICATION_CREDENTIALS`, `GOOGLE_CLOUD_PROJECT`, `VERTEX_AI_LOCATION`
  - PITFALL: polling loop uses `time.sleep` — blocks event loop if called from async context
  - PITFALL: hardcoded GCP project ID is a real production resource
- **Confidence:** HIGH
- **Node recommendation:** NEEDS NODE
- **Reason:** Two auth modes, 3-step preprocessing with N+2 Gemini calls before Veo, blocking polling loop, hardcoded production GCP project, feature flag kill-switch.

## backend/app/services/

- **Owns:** Four orchestration services:
  1. `workflow_executor.py` — compiled Blueprint execution (parallel async DAG via asyncio, `@executor` decorator registry, input resolution, shape conversion, fan-in merging, SSE streaming, Supabase execution log persistence)

  3. `workflow_copilot.py` — NL-to-workflow planner using Gemini. Auto-repair loop (2 LLM attempts + deterministic fallback). Resolves preset IDs from Supabase.
  4. `gradium_voice.py` — Gradium STT (WebSocket streaming) and TTS (HTTP) client wrappers.
- **Does NOT own:** Agent implementation logic, Supabase schema definitions, Gemini client initialization.
- **Spotted contracts:**

  - `execute_workflow_streaming(blueprint) -> AsyncGenerator[str]` (yields SSE JSON events)
  - `compile_workflow(nodes, edges, ...) -> CompilationResult`
  - `plan_workflow_with_copilot(...) -> CopilotPlanResult`
  - `transcribe_audio_bytes(...)`, `synthesize_speech_bytes(...)`
  - Env vars: `GRADIUM_API_KEY`, `GRADIUM_STT_WS_URL`, `GRADIUM_TTS_URL`, `GRADIUM_TTS_VOICE_ID` (default `YTpq7expH9539ERJ`), and more
  - Copilot models: `gemini-2.5-flash` (default tier), `gemini-2.5-pro` (pro tier)
  - Executor imports every agent module directly
- **Confidence:** HIGH
- **Node recommendation:** NEEDS NODE
- **Reason:** Most complex backend directory. Execution engine, compiler, copilot, and voice bridge with distinct responsibilities and non-obvious interactions.

## Cross-Cutting Observations

**Gemini access pattern:** All agents except `image_text_matching` route through `backend/app/llm/gemini.py` and its `GeminiRotationManager`. This is the single most important shared infrastructure.

**Non-Gemini LLM usage:** Two modules use Fireworks AI independently: `image_text_matching` (Qwen VLM via `FIREWORK_API_KEY`) and `audio_transcription` (Whisper via `FIREWORK_API_KEY`). They share the same env var but different endpoints.

**Async-vs-sync inconsistency:** `extract_quotes` is `async def` calling sync `query_gemini`; `ImageTextMatcherVLM` methods are `async def` making blocking Fireworks calls; `generate_video_with_veo` is sync with `time.sleep` polling. None use true async I/O.

**No shared agent contract:** There is no base class, no Pydantic model, and no shared schema across agents. Each defines its own signature independently.

**Manual sync points:** Frontend `nodeRegistry.ts` and `types/blueprint.ts` must be manually kept in sync with backend `models/node_registry.py` and `models/blueprint.py`. `CONNECTED_INPUT_KEYS` in `workflowStore` must also sync with the backend node registry.

**Dead/legacy code:** `scene_detection.py` in `image_extraction`; `ImageMatch.timestamp_score` always 0; `HitlReview` and `ZoomControls` components unused; `hitl.py` and `trigger_job.py` routes largely superseded; `content_parser.py` functions uncalled internally.

### CONFLICT C1: `backend/app/agents/` — Node or No Node?

- **Agents cartographer** found the parent package is empty (no shared code, no base class, no shared schema) and initially recommended NO NODE NEEDED.
- **Cartography plan** proposed this as a Tier 3 node to document "shared patterns across agents."
- **Actual finding:** There ARE no shared patterns in code — but the *absence* of shared patterns is itself architectural knowledge (each agent is standalone, no shared contract). The cross-cutting observations (Gemini routing, async inconsistency, Fireworks deviation) need a home.
- **Decision needed:** Does a node here document what's shared, or should cross-cutting agent concerns live in `backend/app/services/AGENTS.md` (since services are the actual point of integration)?
